# 1C Configurator Theme Engine — Документация проекта

> Обновлено: 2026-07-24 (ночь)
> Статус: **рабочая тёмная тема v5.5** (`ThemeHook3_v55.dll`) + **watcher-автоинжекция**; полный цикл проверен живьём: автоинжекция → тёмная чёткая (`verify_dark_v55_repaint.png`) → выгрузка → светлая чистая (`verify_light_unloaded.png`). Критический урок: **ранняя инжекция (в стартующий процесс) отравляет кэши 1С** — инжектировать только после полной загрузки конфигуратора (см. §6, v5.5)

---

## 1. Проблема и цель

Конфигуратор 1С — нативное Win32-приложение без тёмной темы. Цель проекта: инструмент, позволяющий применять тёмную тему (и произвольные цветовые схемы) к конфигуратору **без модификации файлов платформы**, с возможностью выбора темы пользователем.

**Ограничения, принятые на старте:**
- Не патчить бинарники платформы (лицензия, обновления, нет кастомизации).
- Переживать обновления 1С.
- Давать пользователю выбор темы (JSON-профили — в планах).

## 3. Факты о платформе 1С (установлены экспериментально)

- **1cv8.exe — 32-битный процесс** (даже на 64-битной Windows) → DLL собирается строго x86.
- **Рендер UI — Cairo** из `grphcs.dll`; финальный вывод через `StretchDIBits`/`BitBlt`. Cairo вызывается многопоточно.
- **Справка и синтакс-помощник — WebKit** (`WebKit.dll` в процессе), текст идёт через cairo glyph-вызовы; `ExtTextOutW` там не используется.
- **Окна:** один главный HWND класса `V8Window0.<hash>`, весь UI owner-draw; системные сообщения типа `RedrawWindow` 1С игнорирует — перерисовку надёжно триггерит только ресайз окна (`SetWindowPos`).
- **Часть рендер-модулей подгружается лениво** (F1-справка, диалоги) → IAT-патчинг обязан обрабатывать модули, загруженные после инжекции.
- **Антивирус не блокировал** `CreateRemoteThread`-инжекцию на машине разработчика (не гарантия для других систем).

```

┌──────────────────────────────────────────────┐
│ 1cv8.exe (x86)                               │

│  ThemeHook3.dll (v5.5)                     │

│  │   user32 — поздние модули хукаются сразу│

│  │   • журнал всех патчей (до 4096)          │
│  │   • идемпотентный PatchIAT                │
│  │   • RestoreIAT при DLL_PROCESS_DETACH     │
│  ├─ RescanThread (~3 с): патч новых модулей  │
│  ├─ Хуки Cairo:                              │
│  │   • cairo_set_source_rgb/rgba (+ clip-    │
│  │     контекст: дерево vs редактор)         │
│  │   • cairo_show_glyphs / cairo_show_text   │
│  │     (ghost-fix + ring-suppression дублей, │
│  │      SKIP-ONLY: только пропуск/перекраска)│
│  ├─ Хуки GDI/USER:                           │
│  │   • SetTextColor, ExtTextOutW, DrawTextW  │
│  │     (+FixupGdiTextColor против emboss)    │
│  │   • CreateSolidBrush, FillRect            │
│  │   • GetSysColor (ролевой маппинг по       │
│  │     индексу), GetSysColorBrush (только    │
│  │     безопасные индексы: 5,8,12,13,15,18)  │
```

**Вспомогательные инструменты:**
- `PaletteLog.dll` — сборщик реальной палитры 1С (лог всех цветов с частотами)
- `unload.py` — безопасная выгрузка DLL (FreeLibrary + IAT-restore, процесс выживает)
- `screenshot.py` — скриншоты окна 1cv8 через PrintWindow/GetDIBits
- `tools/poc_find_editor.py` — перечислитель окон/классов процесса 1С

| `#551A8B` | посещённые ссылки | `#B58756` |

**Fallback для неизвестных цветов:** HSL-преобразование — инверсия lightness (фон → L≈0.12, текст → L≈0.82), нейтральные десатурируются (×0.1–0.2, убирает оливковый оттенок), насыщенные акценты сохраняют hue (L=0.65, S×0.85). **Pale-tint правило (v5.1):** цвета с min каналом > 0.55 (бледные тона типа жёлтой шапки F1) классифицируются как фоны, а не акценты — уходят в тёмный фон, а не сохраняют hue.

**Защиты в маппинге:** `IsMappedTarget` guard — уже перекрашенный цвет не маппится повторно (двойной маппинг кистей `CreateSolidBrush`→`FillRect` устранён early-exit'ом).

## 7. Известные ограничения

1. **Синтетический bold WebKit** — смягчён в v5.3–5.4 (dup-skip второго прохода), но как явление движка справки остаётся.
2. **Инжекция только в загрузившийся процесс** — ранняя инжекция (во время старта 1С) отравляет стартовые кэши отрисовки; watcher выдерживает окно «Конфигуратор» + 25 с.
3. Перерисовка после инжекции — только ресайзом окна (гипотеза F не проверена).
4. Антивирусы могут флагать CreateRemoteThread-инжекцию (не подтверждено, но вероятно).
5. Диалоги/формы объектов проверены частично.
6. **Инвариант SKIP-ONLY (v5.4):** нельзя дорисовывать в cairo-контексты и править пиксели кэшированных surface — кэш-отравление переживает выгрузку DLL. Любые будущие фиксы текста — только пропуском дублей или перекраской до отрисовки.
7. **Инвариант ПОЗДНЕЙ ИНЖЕКЦИИ (v5.5):** хуки не должны работать в момент создания стартовых кэшей отрисовки — перекрашенные прогоны запекаются в кэш даже без единой записи пикселей с нашей стороны. Тема применяется только к полностью загруженному конфигуратору.
